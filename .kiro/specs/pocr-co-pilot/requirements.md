# Requirements Document

## Introduction

The POCR Co-Pilot is an AI-assisted embedded reviewer for the Peer Online Course Review (POCR) process used by California Community Colleges (CCC) / California Virtual Campus (CVC). The system integrates directly with Canvas LMS to ingest course content, run automated technical and accessibility diagnostics aligned to the CVC Course Design Rubric, and surface contextual guidance to both faculty and expert peer reviewers. The goal is to reduce per-course human review time from 5–10 hours to under 3 hours by automating compliance and structural checks, freeing reviewers to focus on high-value pedagogical coaching.

---

## Glossary

- **POCR**: Peer Online Course Review — the structured quality-assurance process used by CCC/CVC to certify online courses for the CVC Exchange.
- **CVC Exchange**: The California Virtual Campus cross-enrollment marketplace where badged courses are published for student enrollment across colleges.
- **Canvas**: The Learning Management System (LMS) used by CCC institutions, accessed via REST API.
- **IMSCC**: IMS Common Cartridge — a `.imscc` zip file format for offline course export from Canvas.
- **Rubric**: The Summer 2026/2027 CVC Course Design Rubric, organized into Sections A–D (Course Structure, Pedagogy, Equity, Accessibility).
- **SLO**: Student Learning Objective — a measurable, explicit outcome statement for a course or module.
- **UDL**: Universal Design for Learning — a framework for flexible, inclusive instructional design.
- **AI Policy**: A course-level statement explicitly defining the permitted and prohibited use of generative AI tools by students and faculty.
- **Co-Pilot**: The AI-powered assistant component that provides real-time guidance, rubric-aligned suggestions, and a conversational chat interface.
- **Ingestion_Engine**: The system component responsible for pulling and parsing course content from Canvas API or IMSCC uploads.
- **Diagnostic_Engine**: The system component that runs automated accessibility, structural, and pedagogical checks against ingested course content.
- **Co-Pilot_Dashboard**: The interactive UI summarizing findings by rubric section and providing access to the conversational Co-Pilot.
- **Reviewer**: An expert peer reviewer certified to conduct POCR evaluations.
- **Faculty**: The instructor who authored and is revising the course under review.
- **Report**: The pre-review document generated by the Diagnostic_Engine, summarizing all findings categorized by rubric section.

## Requirements

### Requirement 1: Canvas API Course Ingestion

**User Story:** As a Reviewer or Faculty member, I want to connect a live Canvas course via API token so that the Co-Pilot can access up-to-date course content without manual export.

#### Acceptance Criteria

1. WHEN a user provides a valid Canvas API access token and course ID, THE Ingestion_Engine SHALL retrieve all course content including Pages, Discussions, Assignments, Quizzes, Syllabus, and Modules.
2. IF the provided Canvas API token is invalid or expired, THEN THE Ingestion_Engine SHALL immediately return an error message identifying the authentication failure without attempting content retrieval.
3. IF the Canvas API does not respond within 30 seconds, THEN THE Ingestion_Engine SHALL return an error message describing the connectivity failure and preserve any previously ingested content without modification.
4. WHEN all content types are successfully retrieved without error, THE Ingestion_Engine SHALL store a snapshot of the complete content timestamped to the minute at which retrieval completed, for use by the Diagnostic_Engine.
5. WHEN a user initiates re-ingestion of a previously ingested course, THE Ingestion_Engine SHALL replace the prior content snapshot with the newly retrieved content and update the timestamp to the minute at which the re-ingestion completed.
6. WHEN a Canvas API request succeeds for some content types but fails for others, THE Ingestion_Engine SHALL record the names of each failed content type in an error message, store the successfully retrieved content types in the snapshot, and preserve any prior content for the failed types without modification.

### Requirement 2: Offline IMSCC Export Ingestion

**User Story:** As a Faculty member or Reviewer, I want to upload a `.imscc` course export file so that I can use the Co-Pilot without requiring live API access.

#### Acceptance Criteria

1. WHEN a user uploads a valid `.imscc` file of 500 MB or less, THE Ingestion_Engine SHALL parse the archive and extract all course content types present including Pages, Discussions, Assignments, Quizzes, Syllabus, and Modules; content types absent from the archive SHALL be recorded as empty collections rather than errors.
2. IF the uploaded file exceeds 500 MB or does not conform to the `.imscc` archive format (including files with an incorrect extension regardless of internal structure), THEN THE Ingestion_Engine SHALL return an error message that identifies whether the failure was due to file size or format, and SHALL NOT proceed to parsing.
3. IF the `.imscc` archive is structurally corrupt or does not contain the required `imsmanifest.xml` file, THEN THE Ingestion_Engine SHALL return an error message identifying the specific structural issue and SHALL NOT proceed to the Diagnostic_Engine.
4. WHEN a `.imscc` file is successfully parsed, THE Ingestion_Engine SHALL produce a content representation that contains the same content types, the same item count per content type, and the same metadata fields (title, body, and last-modified date where present) as would be produced by Canvas API ingestion of the same course.
5. WHEN a successfully parsed `.imscc` file is re-ingested without modification, THE Ingestion_Engine SHALL produce a content representation with the same content types, item counts, and metadata fields as the first ingestion of the same file.

### Requirement 3: Embedded Media Caption Detection

**User Story:** As a Reviewer, I want the system to automatically check whether embedded videos have closed captions so that I do not have to manually verify each video element.

#### Acceptance Criteria

1. WHEN course content contains an embedded YouTube video, THE Diagnostic_Engine SHALL query the YouTube Data API to determine whether at least one active closed caption track is available for that video and SHALL record the result as PASSING if at least one track is found.
2. WHEN course content contains an embedded video with a `.srt` or `.vtt` caption track file, THE Diagnostic_Engine SHALL detect the presence of that caption track within 5 seconds and record it as a passing caption check.
3. IF a video element has no detectable closed caption track and no automated transcript flag, THEN THE Diagnostic_Engine SHALL set that video's caption check result to FAILING and record the content location as the module name, page name, and item name.
4. WHEN all video elements in the course have been evaluated, THE Diagnostic_Engine SHALL include in the Report a summary listing each video element with its caption check result (PASSING, FAILING, or INCONCLUSIVE), identified by module name, page name, and item name.
5. IF the YouTube Data API does not respond within 10 seconds, THEN THE Diagnostic_Engine SHALL record the caption check result for affected videos as INCONCLUSIVE and note the API timeout in the Report without modifying any previously recorded results.
6. IF the YouTube Data API returns an error response, THEN THE Diagnostic_Engine SHALL record the caption check result for affected videos as INCONCLUSIVE, include the API error code in the Report, and preserve all caption check results recorded before the error without modification.

### Requirement 4: Heading Structure Accessibility Audit

**User Story:** As a Reviewer, I want the system to automatically verify heading hierarchies across course pages so that I can skip manual heading structure inspections.

#### Acceptance Criteria

1. THE Diagnostic_Engine SHALL parse all HTML content in course Pages and Assignments to extract heading elements (H1 through H6).
2. WHEN a heading hierarchy skips one or more levels (e.g., H1 followed directly by H3), THE Diagnostic_Engine SHALL flag that content item with the content item title and the specific offending heading sequence (e.g., "H1 → H3").
3. WHEN an HTML element uses a `<strong>` or `<b>` tag, or carries inline font-size styling that exceeds the surrounding body text size, in place of a semantic heading element, THE Diagnostic_Engine SHALL flag that content item as a heading structure violation, recording the content item title and the specific element type detected.
4. WHEN a page uses no heading elements, THE Diagnostic_Engine SHALL flag the page as lacking structural navigation cues, recording the page title.
5. WHEN heading structure analysis is complete, THE Diagnostic_Engine SHALL include in the Report a summary of all flagged content items under Section D of the Rubric, where each entry lists the content item title, the violation type (skipped level, pseudo-heading, or no headings), and the offending heading sequence where applicable.

### Requirement 5: Image Accessibility Audit

**User Story:** As a Reviewer, I want the system to automatically detect images with missing or non-descriptive alt text so that I do not have to open every page to find accessibility violations.

#### Acceptance Criteria

1. THE Diagnostic_Engine SHALL parse all HTML content in course Pages, Assignments, and Discussions to identify `<img>` elements.
2. WHEN an `<img>` element has an absent `alt` attribute, THE Diagnostic_Engine SHALL flag that image as failing the alt text check and record its content location (page title and position within the page).
3. WHEN an `<img>` element has an empty `alt=""` attribute without a `role="presentation"` or `aria-hidden="true"` attribute, THE Diagnostic_Engine SHALL flag that image as failing the alt text check and record its content location. WHEN an `<img>` element has an empty `alt=""` attribute together with `role="presentation"` or `aria-hidden="true"`, THE Diagnostic_Engine SHALL record that image as passing the alt text check.
4. WHEN an `<img>` element has an `alt` value that matches a filename pattern (a string containing a file extension such as `.png`, `.jpg`, `.gif`, or `.jpeg`), or consists solely of one of the words "image", "photo", "graphic", "picture", or "img" (case-insensitive), THE Diagnostic_Engine SHALL flag that image as having non-descriptive alt text and record its content location.
5. WHEN image accessibility analysis is complete, THE Diagnostic_Engine SHALL include in the Report a count of passing images, flagged images, and the content location of each flagged image, categorized under Section D of the Rubric.

### Requirement 6: Hyperlink Accessibility and Health Audit

**User Story:** As a Reviewer, I want the system to flag non-descriptive hyperlinks and verify external URL reachability so that I can focus on substantive content review.

#### Acceptance Criteria

1. THE Diagnostic_Engine SHALL extract all hyperlinks from course Pages, Assignments, Discussions, and the Syllabus.
2. WHEN a hyperlink's visible text matches a non-descriptive pattern — specifically "click here", "here", "link", "read more", "more", or a string that begins with "http://" or "https://" — THE Diagnostic_Engine SHALL flag that hyperlink as non-descriptive and record its content location as the page title, content item type, and position within the content.
3. WHEN the Diagnostic_Engine validates an external URL, THE Diagnostic_Engine SHALL issue an HTTP HEAD request; IF the response status code is 4xx or 5xx, THEN THE Diagnostic_Engine SHALL flag that URL as broken and record the status code and content location.
4. IF an HTTP HEAD request does not receive a response within 10 seconds, or if the network connection cannot be established, THEN THE Diagnostic_Engine SHALL flag that URL as unreachable, record the failure reason (timeout or connection failure), and record the content location.
5. WHEN a URL returns a 3xx redirect, THE Diagnostic_Engine SHALL follow the redirect chain up to 5 hops; IF the chain exceeds 5 hops without resolving, THEN THE Diagnostic_Engine SHALL flag that URL as unresolved and record the content location.
6. IF a URL returns a 1xx informational response, THEN THE Diagnostic_Engine SHALL flag that URL for manual review, record the status code and content location, and SHALL NOT classify it as broken or unreachable.
7. WHEN hyperlink analysis is complete, THE Diagnostic_Engine SHALL include in the Report all flagged hyperlinks under Section D of the Rubric, where each entry lists the content location, the flag category (non-descriptive, broken, unreachable, unresolved, or manual review), and the reason for flagging.

### Requirement 7: Course Structure Baseline Checks (Section A)

**User Story:** As a Reviewer, I want the system to automatically verify the presence of required course structural elements so that I can skip Section A checklist items.

#### Acceptance Criteria

1. THE Diagnostic_Engine SHALL check whether a course contains a module whose title includes the word "Orientation" or "Welcome" (case-insensitive) and SHALL flag its absence in the Report under Section A of the Rubric.
2. THE Diagnostic_Engine SHALL check whether a Syllabus item exists and carries a last-modified date within the current academic term or the immediately preceding academic term; IF no Syllabus is present, THE Diagnostic_Engine SHALL flag its absence; IF the Syllabus is present but carries no last-modified date, THE Diagnostic_Engine SHALL flag it as having an unverifiable modification date; IF the Syllabus last-modified date falls outside the two-term window, THE Diagnostic_Engine SHALL flag it as stale.
3. THE Diagnostic_Engine SHALL scan all ingested course documents for at least one statement that explicitly addresses allowed or disallowed use of generative AI tools and SHALL record the AI Policy finding with one of three states: found (with the source document name), not found, or indeterminate (if no documents could be scanned); IF no Syllabus is present, THE Diagnostic_Engine SHALL record the Syllabus absence as a separate finding distinct from the AI Policy finding.
4. THE Diagnostic_Engine SHALL analyze each module to detect Student Learning Objectives, identifying text that contains both an observable action verb (drawn from Bloom's Taxonomy) and a measurable outcome criterion, and SHALL record the count of detected SLOs per module in the Report.
5. IF no SLOs are detected in a module, THEN THE Diagnostic_Engine SHALL flag that module in the Report with a message indicating that measurable SLOs aligned to Bloom's Taxonomy were not detected, without prescribing specific replacement text.
6. THE Diagnostic_Engine SHALL classify each module's title as week-based (title contains a week number or "Week" label) or topic-based (title does not contain a week or date reference); IF fewer than 80% of modules share the same classification, THEN THE Diagnostic_Engine SHALL flag the course as having an inconsistent module structure in the Report.

### Requirement 8: Equity and Inclusion Scan (Section C)

**User Story:** As a Reviewer, I want the system to surface equity and inclusion opportunities in course content so that I can provide more targeted coaching to faculty.

#### Acceptance Criteria

1. WHEN analyzing course content, THE Diagnostic_Engine SHALL identify content items that lack representation of diverse socio-economic, cultural, or lifestyle perspectives and SHALL flag them as equity review candidates in the Report.
2. WHEN analyzing course materials and resources, THE Diagnostic_Engine SHALL flag content items where 100% of cited or featured sources, authors, subjects, or represented voices share the same visible demographic characteristic (e.g., all sources authored by individuals from a single national origin, gender, or ethnicity as inferable from names and context).
3. WHEN analyzing assignment prompts, THE Diagnostic_Engine SHALL flag prompts that assume a single cultural context or require access to resources not universally available to students, including physical materials, paid subscription services, or location-specific facilities.
4. IF a content item cannot be analyzed for equity signals due to unreadable format, missing content, or unsupported media type, THEN THE Diagnostic_Engine SHALL record that item as unevaluated in the equity findings and include the item title and reason in the Report without counting it as a pass or fail.
5. WHEN equity analysis is complete, THE Diagnostic_Engine SHALL include in the Report all flagged content items under Section C of the Rubric, with at least one specific suggestion for broadening representation per flagged item.

---

### Requirement 9: AI Policy Detection

**User Story:** As a Reviewer, I want the system to automatically check whether the course syllabus contains an explicit AI usage policy so that I can confirm compliance with the 2026/2027 Rubric requirement.

#### Acceptance Criteria

1. WHEN the Syllabus or any ingested course document is present, THE Diagnostic_Engine SHALL scan its text for at least one statement that explicitly addresses either allowed or disallowed use of generative AI tools by students or faculty.
2. WHEN an AI policy statement is detected in any course document, THE Diagnostic_Engine SHALL extract up to 1,000 characters of the relevant text, record the source document name, and mark the AI Policy check as passing in the Report.
3. IF no AI policy statement is detected in any ingested course document, THEN THE Diagnostic_Engine SHALL mark the AI Policy check as failing, include a template suggestion of no more than 500 characters for an AI policy statement in the Report, and SHALL NOT modify the pass/fail result of any other diagnostic check.
4. WHEN the AI policy check is complete, THE Diagnostic_Engine SHALL categorize the result under Section A of the Rubric in the Report.
5. IF a course document cannot be scanned due to an unsupported format or read error, THEN THE Diagnostic_Engine SHALL mark the AI Policy check result for that document as indeterminate, include an error message identifying the document and the reason it could not be scanned, and exclude the unscanned document from the pass/fail determination.

### Requirement 10: Pre-Review Report Generation

**User Story:** As a Reviewer, I want to receive a structured pre-review report before beginning my review so that I can immediately focus on high-value pedagogical coaching instead of compliance checking.

#### Acceptance Criteria

1. WHEN all Diagnostic_Engine checks are complete, THE Co-Pilot_Dashboard SHALL generate a Report that categorizes all findings under their corresponding Rubric sections (A, B, C, D).
2. THE Report SHALL include a summary count of passing checks, flagged items, and inconclusive items for each Rubric section.
3. THE Report SHALL include the specific content location (module name, page title, or item name) for every flagged item.
4. THE Report SHALL include a severity classification for each flagged item, where Critical indicates the item fails a mandatory Rubric requirement, Advisory indicates it deviates from a recommended practice, and Informational indicates it warrants human review but is not a rubric failure.
5. WHEN all diagnostic checks are complete, THE Co-Pilot_Dashboard SHALL display the Report to the user within 30 seconds of the final diagnostic check completing; IF the Report is not ready within 30 seconds, THE Co-Pilot_Dashboard SHALL display a progress indicator and present the Report as soon as it becomes available, up to a maximum of 5 minutes after the final check completes.
6. WHEN the Report is displayed, THE Co-Pilot_Dashboard SHALL make it exportable as a PDF or HTML document; IF the export fails, THEN THE Co-Pilot_Dashboard SHALL display an error message and preserve the Report in the dashboard without interrupting the user's review session.

### Requirement 11: Contextual Inline Co-Pilot Suggestions

**User Story:** As a Faculty member reviewing findings, I want to see non-intrusive, inline suggestions tied to specific content items so that I understand exactly what to fix and how.

#### Acceptance Criteria

1. WHEN a flagged item is selected in the Co-Pilot_Dashboard, THE Co-Pilot SHALL display a contextual suggestion panel describing the issue, its rubric section reference, and at least one concrete remediation action.
2. THE Co-Pilot SHALL present suggestion panels as floating overlays or sidebar panels positioned so that the primary content navigation area (module list and page content) remains fully visible and interactive.
3. WHEN a Faculty member dismisses a suggestion, THE Co-Pilot SHALL record the dismissal and SHALL NOT display the same suggestion for the same content item again during the active session; IF the dismissal recording fails, THE Co-Pilot SHALL continue to display the suggestion until the dismissal is successfully confirmed.
4. WHEN a content item is updated and the next ingestion cycle completes, THE Co-Pilot SHALL mark any suggestion associated with that content item as resolved if the previously flagged condition is no longer detected.

---

### Requirement 12: Conversational Creative Partner Mode

**User Story:** As a Faculty member or Reviewer, I want to chat with the Co-Pilot to brainstorm rubric-aligned improvements so that I can generate ideas without leaving the review interface.

#### Acceptance Criteria

1. THE Co-Pilot SHALL provide a conversational chat interface accessible from the Co-Pilot_Dashboard without requiring navigation away from the current view.
2. WHEN a user submits a message in the chat interface that can be answered without multi-step rubric analysis, THE Co-Pilot SHALL respond within 10 seconds; WHERE a question requires complex rubric analysis or generates a multi-part response, THE Co-Pilot SHALL provide a complete response and is not subject to the 10-second target.
3. WHEN a user asks the Co-Pilot to reword a Student Learning Objective, THE Co-Pilot SHALL generate at least two alternative phrasings, each using an observable action verb from Bloom's Taxonomy and preserving the core learning intent of the original.
4. WHEN a user asks the Co-Pilot to suggest equity-focused scenarios for an assignment, THE Co-Pilot SHALL generate at least two scenario options, each set in a distinct cultural or socio-economic context.
5. WHEN a user submits a message that does not relate to course design, the CVC Rubric, or POCR, THE Co-Pilot SHALL acknowledge that the topic is outside its scope and provide a prompt redirecting the conversation to a rubric-relevant topic.
6. THE Co-Pilot SHALL retain all messages exchanged within the active session so that follow-up questions can reference prior messages without requiring the user to restate context; session history SHALL be cleared when the session ends.

### Requirement 13: Four-Phase POCR Workflow Support

**User Story:** As a program administrator, I want the system to support the discrete four-phase POCR workflow so that Faculty and Reviewers interact with the Co-Pilot at the appropriate phase of the process.

#### Acceptance Criteria

1. THE Co-Pilot_Dashboard SHALL present four distinct views corresponding to Phase 1 (Faculty Course Prep), Phase 2 (Automated AI Pre-Audit), Phase 3 (Expert Peer Review), and Phase 4 (Author Revisions), with the active phase clearly indicated.
2. WHEN a course is advanced to Phase 2, THE Diagnostic_Engine SHALL automatically initiate ingestion and all diagnostic checks and SHALL complete the Report before the course can be advanced to Phase 3.
3. WHILE a course is in Phase 3, THE Co-Pilot_Dashboard SHALL display the Report and Co-Pilot chat interface to the Reviewer with full access, and SHALL restrict Faculty access to read-only viewing of the dashboard with no ability to edit or dismiss findings.
4. WHILE a course is in Phase 4, THE Co-Pilot_Dashboard SHALL present Faculty with the Reviewer's annotated findings and SHALL enable Faculty to mark each individual finding as addressed.
5. WHEN all Phase 4 findings are marked as addressed, THE Co-Pilot_Dashboard SHALL generate a completion summary listing all addressed findings and their resolutions, and SHALL set the course status to "Ready for CVC Exchange Badging Review."

---

### Requirement 14: Review Time Reduction Target

**User Story:** As a program administrator, I want the system to measurably reduce per-course review time so that we can scale POCR throughput without increasing reviewer headcount.

#### Acceptance Criteria

1. WHEN ingestion is complete, THE Diagnostic_Engine SHALL complete all automated checks for a course containing up to 200 content items within 5 minutes of ingestion completion, measured from the moment the Ingestion_Engine signals completion to the moment the final diagnostic check result is recorded.
2. WHEN a Reviewer begins Phase 3 using the Co-Pilot, THE Co-Pilot_Dashboard SHALL present the full pre-audited Report with all Diagnostic_Engine findings so that no check covered by the Diagnostic_Engine appears as an open item requiring manual reviewer action.
3. WHEN a review session ends, THE Co-Pilot_Dashboard SHALL display an estimated time saved for that session, calculated as the sum of the predefined average manual check duration for each Diagnostic_Engine check type that produced a passing or flagged result during the session.
